# hplc-design-space スキル description 評価レポート

実施日: 2026-06-27 / ツール: skill-creator の description 最適化ループ（`scripts/run_loop.py`）
モデル: claude-opus-4-8 / 設問: `trigger-eval.json`（起動すべき10件・起動すべきでない10件）

## やったこと

スキルが「正しい場面で起動し、似て非なる場面では起動しない」かを定量評価するため、
20件のトリガー設問を用意し、`claude -p` で各設問を3回ずつ実行して起動率を測定。
description を反復改善する自動ループを最大5反復で回した。

## 結果（数値は環境都合で参考値）

| 反復 | description | precision | recall | accuracy |
|---|---|---|---|---|
| 1 | 現行版 | 100% | 0% | 50% |
| 2 | 改善案A（口語＋除外明示） | 100% | 0% | 50% |
| 3 | 改善案B（簡潔版） | 100% | 0% | 50% |

反復3の改善ステップで `claude -p` が一時的に exit 1 となりループ終了。

## 重要な所見：自動トリガー評価はこの環境では信用できない

全反復・全 description で **recall=0%**（「起動すべき」設問が1件も起動しない）。
一番ど真ん中の設問（10-ジンゲロールが夾雑ピーク2本に挟まれ、T/φ/F を最適化してデザインスペースを出す）
すら 0/3。これは description の良し悪しではなく**評価ハーネスの限界**を示す：

- `run_eval.py` はスキルを `.claude/commands/` に登録して `claude -p` に起動させる方式だが、
  この Claude Code 版（2.1.195）では登録したスキルが自動起動候補（available_skills）に乗らず、
  どんな設問でも発火しない。
- 結果、ポジティブ全滅・ネガティブ全 PASS（何も起動しないので自明に PASS）。
  信号が平坦なため最適化が機能しない。

→ **数値を根拠に description を決めることはできない**と判断。

## それでも採用した改善（ベスト practice ベース・数値未実証）

ループ生成案と手動レビューが一致した、定石どおりの改善を description に適用：

1. **口語トリガーの追加**（「ピークが重なって分けられない」など、ユーザーが実際に言う表現）
2. **除外スコープの明示**（検量線・定量、システム適合性、前処理最適化、LC-MS 同定、
   溶出 f2、GC、抗体グラジエント／ペプチドマッピング＝近接するが別作業）
3. やや push 気味の言い回し（undertrigger 対策）

これらは起動精度向上の定石だが、本環境では定量実証できていない点に留意。
