## プロジェクト概要

- **目的**：漢方中の 10-gingerol を HPLC で頑健に分離・定量するため、カラム温度 T・流速 F・溶媒比 φ（水:ACN）の3条件を統計的に最適化し、デザインスペースを3Dで可視化する
- **現行条件**：T=75℃ / F=0.6 mL/min / 水:ACN=60:40（φ=40% ACN）アイソクラティック
- **問題**：目的ピーク(TP)が前後の夾雑ピーク(IP1, IP2)に挟まれ、条件ブレで分離度が落ちるリスク。クロスオーバーあり。

---

## ピークの命名

| 略称 | 意味 | 現行条件での溶出順 |
|------|------|------------------|
| IP1 | Interfering Peak 1（夾雑ピーク1） | 1番目 |
| TP | Target Peak = 10-gingerol（目的ピーク） | 2番目 |
| IP2 | Interfering Peak 2（夾雑ピーク2） | 3番目 |

## 確定した設計判断

### 指摘1：流速 F のモデル上の扱い（確定）

- k（保持係数）は T と φ のみで決まる。F は保持に影響しない
- 保持モデル：`ln k = a + b/T[K] + c·φ`（必要なら +d·φ²） × 3本（IP1, TP, IP2）
- 保持時間の復元：`t_R = (V_m / F) × (1 + k)`
- 幅モデル：`W(T, φ, F)` × 3本（F が主役。理論段数 N を介す案が安定）
- V_m はカラム固有定数。不保持物質（ウラシル等）を現行条件で1回注入して確定

### 指摘2：デザインスペースの定義（確定）

- **合格基準**：`min{Rs1, Rs2} ≥ 2.0`
- **最適条件の選び方**：デザインスペース（合格領域）の中で、**境界からの最短距離が最大になる点**（最大余裕点）を採用
- 根拠：ICH Q8。条件ブレに対して最もロバストな点を選ぶ

### 指摘3：Rs1/Rs2 のクロスオーバー対応（確定）

- Rs は溶出順でなく**化合物の同一性**で定義する
- `Rs1 = 2·|t_R(TP) − t_R(IP1)| / (W_TP + W_IP1)`
- `Rs2 = 2·|t_R(TP) − t_R(IP2)| / (W_TP + W_IP2)`
- 絶対値 |Δt| を使うため、溶出順が入れ替わっても Rs は常に正しく計算される

### 指摘4：探索範囲と合格条件（確定）

**カラム情報**
- Waters ACQUITY UPLC BEH C8, 130Å, 1.7µm, 2.1mm × 100mm
- 耐久上限 90℃ / 実用推奨上限 80℃（ACN 沸点 82℃ への安全マージン）

**探索範囲（暫定）**

| 因子 | 下限 | 中水準 | 上限 |
|------|------|--------|------|
| T（温度） | 40℃ | 50℃ | 60℃ |
| φ（ACN 比率） | 38% | 45% | 52% |
| F（流速） | 0.4 mL/min | 0.6 mL/min | 0.8 mL/min |

- F の水準範囲は `t_R = V_m(1+k)/F` の反比例から計算。V_m・k（中水準）の実測値が出たら `F_low = V_m(1+k)/tR_high`、`F_high = V_m(1+k)/tR_low` で更新する
- 各因子の目標効果量：t_R ± 1分（端から端まで変えたときの変化量）
- 現行条件（75℃ / 40% ACN）より低温・高 ACN 側にシフト

**合格条件（デザインスペース判定に使う）**

| 条件 | 閾値 | 備考 |
|------|------|------|
| Rs = min{Rs1, Rs2} | ≥ 2.0 | デザインスペースの合否 |
| t_R(TP) | ≤ 7.5分 | デザインスペースの合否。TP固有の分析時間上限 |
| max(t_R 全3本) | ≤ 10.0分 | **デザインスペースの合否には含めない**。これは「データ取り段階の洗浄前制約」であり、運用条件として課したい場合のみ任意で適用（既定 null） |

**水準範囲の最終確定フロー**
1. 指摘7の実測後にモデルをフィット
2. 各合格条件の等高線を T-φ-F 空間に描く
3. 全条件を満たす領域をデザインスペースとして確定
4. 感度分析で各因子の t_R への効果量を確認し、水準範囲を微調整して D最適追加実験へ

### 指摘5：実験本数と逐次 augment（確定）

**実験条件**
- 1注入 = 22分（洗浄込み）
- 実験本数の上限：1日あたり50本

**逐次デザイン（別日 + ブロッキング）**

| 日 | 内容 | 本数 |
|----|------|------|
| Day 1 | CCD：頂点 2³=8 + 軸上点 6 + 中心点 6 | 20本 |
| Day 2 | D最適 augment + 中心点3本（橋渡し） | 必要分（1日50本枠内） |

- CCD は中心点 n=6 でスタート（純誤差の自由度5・湾曲検定の安定性を確保）
- 中心点はこれ以上増やしても因子空間の情報は増えない（純誤差推定は頭打ち）
- モデルが十分なら D最適 augment はスキップ可

**ブロッキング（日間変動の補正）**
- CCD と D最適を別日に実施 → 日間変動（移動相ロット・室温・装置状態）が混入する
- 対策：モデルに日を表す項を追加 `ln k = a + b/T + c·φ + δ·(day)`
- 各日に中心点を入れ、日間オフセットを推定して補正
- Day2 冒頭の中心点3本が Day1↔Day2 の橋渡し（ズレ量の測定）

**自由度の考え方（本数の妥当性）**
- 1注入で3ピーク全部の t_R・W_h が測れる → 自由度は「総本数」でなく「モデルごと」に数える
- モデルは独立6本（保持3＋幅3）。各モデルが全実験本数を観測点に使える
  - 保持（1ピーク）：`ln k = a + b/T + c·φ + d·φ² + e·φ/T`＝5パラメータ → Day1 20本で残差自由度15
  - 幅（1ピーク）：`H = A + B/u + C·u`＝3パラメータ → 残差自由度17
- 保持は実質「(T,φ)の2次曲面」。3因子CCD=20本は2次モデル当てに教科書的に最適サイズ
- 中心点6本＝純誤差自由度5 → ラック・オブ・フィット検定も可能
- 結論：Day1の20本でモデル当ては十分。Day2のD最適は自由度不足の補填でなく
  「デザインスペース境界の予測精度向上・モデル再確認・日間差測定」が目的（モデルが十分なら省略可）

### 最大余裕点の「崖」の定義（確定）

- 最大余裕点＝合格領域の中で「最寄りの崖まで最も遠い点」。崖は2種類で測る：
  1. **合格条件を破る不合格点**（Rs不足・時間超過の境界）
  2. **因子範囲の壁（探索範囲の端）** — ここも崖として扱う
- 理由：範囲の端ぎりぎりの点は、条件が少し上振れすると**未検証域への外挿**になる。
  端を崖に含めると推奨点が範囲中央寄りに引き戻され、ICH Q8 の「ロバストな運転点」の趣旨に合う
- 実装：各因子を範囲幅で 0〜1 に正規化し、壁までの距離 `min(x, 1−x)` と
  不合格点までの距離の小さい方を「余裕」とする（`scripts/04_optimize.py`）

### 指摘7：モデル初期化に必要な実測（確定）

**ピーク幅の定義**
- 半値幅 W_h を採用。理論段数へは `N = 5.54·(t_R/W_h)²` で変換

**V_m（カラム死容量）の決め方**
- 第一選択：幾何推算 `V_m ≈ 空隙率 × π r² L ≈ 0.68 × 0.346 ≈ 0.24 mL`（2.1×100mm, 空隙率0.68）
  - 注意：`t_0 = V_m/F`（時間）と `V_m = t_0·F`（体積）は別物。V_m は体積なので流速で割らない
- 厳密化する場合：不保持物質（ウラシル等）を1回注入し `t_0` を実測 → `V_m = t_0·F`

**必要な実測リスト**
| 区分 | 項目 | 回数 |
|------|------|------|
| カラム定数 | L=100mm, 内径2.1mm, 粒径1.7µm（データシート） | — |
| カラム定数 | V_m（幾何推算0.24 or ウラシル実測） | 0〜1回 |
| 現行条件アンカー | 75℃/40%/0.6 での IP1/TP/IP2 の t_R・W_h | 1回（任意・サニティ用） |
| DoE 本番 | 各条件で3ピークの t_R・W_h | CCD計画ぶん |

**純誤差について（Q2の結論）**
- 現行条件での反復測定は不要。純誤差は CCD の中心点6本で取得する
- 現行条件の実測は「モデルが既知点を通るかのサニティチェック／初期値の当たり」用で、取れれば1回で十分（必須でない）

### Excel×Python 連携方式（確定）

**方式：Python in Excel ＋ ハイブリッド**

ユーザーは Python in Excel（Microsoft 純正、セル内 Python）で作業したい。
ただし Python in Excel には制約があるため、以下のハイブリッド構成にする。

**Python in Excel の制約（確認済み）**
- 自作モジュールを import 不可 → コードはセルに貼り付ける形
- ローカルファイルにアクセス不可（クラウドサンドボックス）→ plotly HTML をファイル出力できない
- 既定で使える：numpy / pandas / scipy / statsmodels / matplotlib（フィット・最適化は問題なし）

**役割分担**
| 工程 | 実行場所 |
|------|---------|
| データ入力・モデルフィット・最適化・デザインスペース判定 | Python in Excel（セル）|
| 簡易プレビュー（静止画 3D / 2D 等高線） | Python in Excel（matplotlib）|
| 最終的な対話的 3D デザインスペース（plotly HTML、回転可・報告用） | 通常 Python 環境で scripts/05 を1回走らせる |

**実装方針（重要）**
- `scripts/` の中核ロジックは **numpy/scipy/statsmodels/pandas のみ**で書く
  → 同じコードを「通常環境でモジュール import」「Python in Excel のセルに貼付」の両方で使える（二重管理回避）
- plotly は最終 3D 出力(scripts/05)専用。Excel 側は matplotlib のみ

**フロントエンド：Excel と Streamlit の両対応（確定）**
- 中核ロジックが numpy 系のみなので、2つの入口を無理なく載せられる
| 入口 | 用途 | 3D |
|------|------|----|
| Python in Excel | Excel に馴染んだ作業・手元計算 | matplotlib 静止画 |
| Streamlit Web アプリ | ボタン操作・報告で見せる対話的可視化 | plotly（回転可）|
- 両者は同じ `scripts/` の関数を呼ぶだけ。ロジックは一箇所に集約
- Streamlit 起動：手元PCで `streamlit run app.py`（または Community Cloud に配置）
